# fix-5c1ad3a557d400d8 — Callback handlers `join:round` and `start:round` — missing username requirement

**Weight:** 0.0000 (share of project budget)
**Reward:** 0 ELIM

The `/join` command in `E6T1.ts:31-34` properly rejects users without a Telegram username. However, the `join:round` callback handlers in both `E1T1.ts:28-36` and `E9T3.ts:27-35` pass `ctx.from.username` (which can be `undefined`) directly to `repository.joinRound()` without any username check. In `redis-repository.ts:441`, `usernameKey()` falls back to `String(user.id)`, so users without usernames get silently keyed by numeric ID — creating an inconsistency where the same user could be identified by two different keys (username vs. numeric ID) depending on how they interact.

Similarly, `E9T4.ts:21` has `start:round` using `ctx.from.username ?? String(ctx.from.id)` — another fallback to numeric ID without a proper username check. If the group creator originally joined with a username, calling `start:round` without a username would match against a different string (the numeric ID), causing a misleading `not_creator` error instead of a clear "set a username" message.

These handlers are part of the round join/elimination logic that E8T3's spec says should "reference usernames instead of user IDs" and "ensure proper error handling for missing usernames" (per E8T2's spec for the related commands).
